## 동기

> 우테코 오픈 미션은 2주간 자신이 성장할 수 있는 미션을 하는 것입니다.  
그래서 저는 평소 만들고 싶었던 웹을 **바닐라 자바스크립트로 구현 후 리액트로 마이그레이션** 해보려고 합니다.


## 평소 만들고 싶었던 웹이란

저는 인터넷 방송을 즐겨보는데, 그 중 파카라는 스트리머의 방송을 주로 봅니다. 주로 리그오브레전드 방송을 주로 하지만,   본업인 `리그오브레전드` 보다 가끔 하는 `종합 게임`이 더 재밌게 느껴졌었습니다.

하지만 "치지직"은 방송 켜지는 것에 대한 알람만 있고, 카테고리 변경시 알람 기능이 없었습니다.  
그래서 저는 제 필요에 의해 **카테고리 변경 시 알림을 받을 수 있는 웹**을 구축하려고 합니다.

## 프로젝트 개요 및 설계 의도
저는 단순한 기능 구현이 아니라, 프레임워크가 없는 환경에서 발생하는 문제들을 순수한 `JavaScript` 능력으로 해결해보며 언어의 이해와 프레임 워크가 나오게 된 이유를 몸으로 느껴보기 위해 선택했습니다.

따라서 구현을 시작하기 전, 이번 프로젝트로 이룰 수 있는 것들을 정리했습니다.  

### 왜 `Vanilla JS`인가?  
기술을 조사하던 중 현대 프론트엔드는 `CSR`-`SSR`-`RSC` 등의 기술 변화를 겪고 있다는 걸 확인했습니다.

이를 보고 프레임워크는 언제든지 바뀔 수 있다고 생각했기 때문에 `JavaScript` 언어의 기초를 다지는 것이 최우선 목표라고 생각했습니다.  

- 1단계 (Vanilla JS): 상태 관리, 이벤트 처리 등 핵심 비즈니스 로직을 JavaScript SPA로 구축하여, '프레임워크가 없으면 발생하는 문제'를 직접 경험합니다.

- 2단계 (마이그레이션): 1단계의 코드를 React와 TypeScript로 전환하며, **제가 직접 구현했던 패턴(옵저버)**이 프레임워크에서는 Hooks 같은 형태로 어떻게 추상화되는지 비교 분석하는 데 초점을 맞춥니다.

### 핵심 디자인 패턴을 활용한 문제 해결
`Vanilla JS`로 개발하면서 필연적으로 발생하는 유지보수성과 확장성 문제를 해결하기 위해 다음 `OOP 패턴`들을 도입했습니다.

- **옵저버 패턴 (Observer Pattern)**: 카테고리 변경이라는 이벤트를 중앙 NotificationManager가 발행하면, 현황판 UI와 크롬 알림 시스템이 각자의 역할을 수행하도록 연결했습니다. 이를 통해 상태(데이터)와 뷰(화면)의 결합도를 낮춥니다.

- **어댑터 패턴 (Adapter Pattern)**: 외부 치지직 API의 복잡하고 불규칙한 응답을 애플리케이션 내부에서 다루기 쉬운 Streamer 모델로 변환했습니다. 외부 API 구조가 바뀌더라도 내부 로직 변경을 최소화하기 위함입니다.

- ***싱글턴 패턴 (Singleton Pattern)**: 앱 전체에서 오직 하나만 존재해야 하는 NotificationManager나 AppRouter에 적용하여, 어디서든 동일한 인스턴스로 접근할 수 있도록 중앙 관리 지점을 명확히 했습니다.

### 주요 기술 스택
프론트엔드: Vanilla JavaScript

백엔드: Node.js (Express)

API: 치지직 공식 API

확장: TypeScript (코드 안정화), React (UI 전환)
